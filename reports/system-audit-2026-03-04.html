Classifier Returns confidence: 0 — All Requests Default to Ron

The LLM classifier (Haiku 4.5) wraps its JSON response in markdown code fences (```json ... ```), but the extraction regex /\{[^}]*\}/ fails to parse it. The catch block returns { expert: "ron", confidence: 0.0, reason: "Classification error" }.

Impact: Every request routes to Ron regardless of content. Your multi-domain question (contracts + engagement + research) should have been split across Ron (data), and Leslie (research), but instead Ron handled everything solo.

Root cause: classifier.ts:40 — regex /\{[^}]*\}/ can't handle code-fenced or multiline JSON. Haiku's response text is ```json\n{"expert":"ben",...}\n```, and the regex fails on the fence markers.

Fix applied: Updated regex to strip code fences first, then use /\{[\s\S]*\}/ for multiline matching.

Memory Plugin Not Loading in Expert Containers

All six expert containers log: [plugins] memory slot plugin not found or not marked as memory: memory-core. OpenClaw can't find the memory-core plugin, meaning the dedicated recall/remember tools aren't available.

Ron works around this by using the exec tool (shell commands) to run bd recall and bd remember directly. This is the SOUL.md fallback path and it works — but it's less efficient than native tool calls.

Additionally, the tools profile reports: tools.profile (coding) allowlist contains unknown entries (apply_patch, memory_search, memory_get, cron). These tool names are referenced in config but don't exist as registered plugins.

Beads State: 0 Open Issues Across All Experts

Every expert container reports Beads OK: 0 open memories. Despite Ron claiming to "save a memory bead" in both responses, no beads persist in any container's Dolt database. This could mean:

1. bd sync is deprecated — containers log bd sync is deprecated. Use 'bd dolt push' and 'bd dolt pull' instead. The entrypoint still calls bd sync, which may not actually commit data.

3. Ron's bd commands may silently fail — the exec tool logs show calls completing but we can't see the output.

Design Note

The current architecture routes to ONE expert per request. Even with the classifier fixed, a multi-domain question like "contracts + engagement + research" will go to whichever expert scores highest. To truly split across experts, the gateway would need a multi-route capability — a future enhancement worth considering.

Recommendations

▶
1

Rebuild Gateway to Deploy Classifier Fix

The regex fix in classifier.ts has been applied locally. Run docker compose up -d --build chris to deploy. This will restore proper expert routing based on message content.

2

Investigate Memory Plugin Registration

The memory-core plugin isn't being found by OpenClaw. Either the plugin path in the entrypoint config needs updating, or the plugin needs to be registered differently. For now, the exec fallback works but native tools would be more efficient.

3

Update Entrypoint to Use bd dolt push/pull

Replace bd sync calls with the non-deprecated bd dolt push and bd dolt pull commands. The deprecation warning suggests bd sync may not actually persist data.

4

Consider Multi-Expert Routing for Complex Questions

The current architecture picks one expert per request. For questions that span multiple domains, the gateway could fan out to multiple experts and aggregate responses. This is a significant architectural change but would improve answer quality for cross-domain questions.
